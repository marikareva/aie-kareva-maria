- Как был организован `train / validation / test`: данные разбиты по времени в соотношении 60% train, 20% validation, 20% test.
- Какие подходы сравнивались (`B1`, `B2`, `B3`, `R1`):

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: Данные были отсортированы по времени в хронологическом порядке, после чего выполнено последовательное разбиение:
  - Первые 60% наблюдений отнесены к обучающей выборке (train)
  - Следующие 20% — к валидационной выборке (validation)
  - Последние 20% — к тестовой выборке (test)
- Какие доли или интервалы использовались: 
  - Train: 60% данных (2592 наблюдения) — период с 2025-01-01 00:00:00 по 2025-04-18 23:00:00
  - Validation: 20% данных (864 наблюдения) — период с 2025-04-19 00:00:00 по 2025-05-24 23:00:00
  - Test: 20% данных (864 наблюдения) — период с 2025-05-25 00:00:00 по 2025-06-29 23:00:00
- Почему `random split` здесь некорректен: Random split некорректен для временных рядов, потому что значения ряда зависят от предыдущих наблюдений (автокорреляция), и случайное перемешивание данных разрушает эту временную структуру, позволяя модели использовать информацию из будущего при обучении, что приводит к утечке данных и искусственному завышению качества прогноза, в то время как в реальных задачах мы всегда прогнозируем будущее только на основе прошлого.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались:
  - `lag_1` — значение за предыдущий час
  - `lag_7` — значение за 7 часов назад (учитывает суточную сезонность, так как в сутках 24 часа, но 7 часов — это примерно треть суток)
  - `lag_14` — значение за 14 часов назад
- Какие rolling-признаки использовались:
  - `rolling_mean_7` — скользящее среднее за последние 7 часов
  - `rolling_std_7` — скользящее стандартное отклонение за последние 7 часов
  - `rolling_mean_14` — скользящее среднее за последние 14 часов
  - `rolling_std_14` — скользящее стандартное отклонение за последние 14 часов
- Какие календарные признаки использовались:
  - `hour` — час дня (0-23)
  - `day_of_week` — день недели (0-6, где 0 — понедельник)
  - `month` — месяц (1-12)
  - `day_of_month` — число месяца (1-31)
  - `hour_sin`, `hour_cos` — циклическое кодирование часа (sin/cos трансформация для учета цикличности 24 часов)
  - `day_of_week_sin`, `day_of_week_cos` — циклическое кодирование дня недели (sin/cos для учета цикличности 7 дней)
- Как обрабатывались пропуски после построения лагов/rolling: При создании лаговых признаков (shift) и rolling статистик первые 14 строк каждой выборки содержали NaN (из-за lag_14, который требует 14 предыдущих значений). Эти строки были удалены с помощью метода `.dropna()` для каждой выборки отдельно. В результате из train удалено 14 строк (осталось 2578), из validation — 14 строк (осталось 850), из test — 14 строк (осталось 850).
- Использовалось ли масштабирование и где именно оно обучалось: Да, для моделей B3 (Ridge Regression) и R1 (GRU) использовалось масштабирование признаков и целевой переменной с помощью `StandardScaler`. Масштабирование обучалось **только на train данных** (метод `.fit()`), а затем применялось к validation и test с помощью `.transform()` для предотвращения утечки информации.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): Наивный baseline, где прогноз на все горизонты равен последнему известному значению из обучающей выборки (122.68). Не требует обучения, не использует признаки. Служит нижней границей качества.

- B2 (`moving-average`): Baseline на основе скользящего среднего, где прогноз равен среднему арифметическому последних 7 значений из обучающей выборки (141.18). Не требует обучения, использует только окно фиксированного размера. Дополнительно проведен эксперимент с разными размерами окна (3, 7, 14, 24), лучший результат на validation показало окно 24 (MAE=14.96), но для основного эксперимента оставлено окно 7 согласно заданию.

- B3 (`ridge-lag-features`): Линейная модель с L2-регуляризацией (Ridge regression), обученная на 15 признаках: лаги (1, 7, 14), rolling статистики (среднее и std для окон 7 и 14), календарные признаки (час, день недели, месяц, день месяца) и их циклическое кодирование. Подбор гиперпараметра alpha осуществлялся по валидационной MAE в диапазоне [0.01, 0.1, 1.0, 10.0, 100.0]. Лучшее значение alpha=0.01. Масштабирование признаков выполнено с помощью StandardScaler, обученного только на train. Модель показала лучший результат среди всех экспериментов.

- R1 (`gru-forecast`): Рекуррентная нейронная сеть (GRU) для прогнозирования следующего значения на основе оконного представления ряда. Модель принимает на вход последовательность из 24 предыдущих наблюдений (window_size=24) и предсказывает одно следующее значение. Архитектура: 2 слоя GRU с hidden_size=128, dropout=0.2 для регуляризации, полносвязный слой на выходе. Масштабирование признаков и целевой переменной выполнено с помощью StandardScaler, обученного только на train. Модель обучалась на 2554 последовательностях, валидация на 826 последовательностях.

Дополнительно:

- Основная метрика выбора лучшей модели: MAE (Mean Absolute Error) на валидационной выборке
- Дополнительные метрики: RMSE (Root Mean Square Error), MAPE (Mean Absolute Percentage Error)
- `window_size` для `GRU`: 24 
- Batch size: 64
- Максимальное число эпох: 100
- Optimizer: Adam
- Learning rate: 0.001
- Как сохранялась лучшая модель: В процессе обучения на каждой эпохе вычислялась MSE на валидационной выборке. Лучшая модель (с минимальной валидационной потерей) сохранялась с помощью `model.state_dict()`. После завершения обучения лучшая модель была сохранена в файл `artifacts/best_gru.pt`, а её конфигурация (архитектура, гиперпараметры, список признаков) — в `artifacts/best_gru_config.json`.

## 7. Анализ

Наивный baseline (B1) оказался слабым, так как он игнорирует всю историческую информацию, просто повторяя последнее наблюдение. При наличии выраженного тренда и сезонности такой подход неизбежно дает высокую ошибку, особенно на тесте, где тренд усиливается (MAE выросла с 24.26 на validation до 32.28 на test).

Moving average (B2) с окном 7 дал значительное улучшение (MAE 15.27), так как усреднение последних значений сглаживает шум и учитывает краткосрочную динамику. Однако этот подход все еще не улавливает суточную сезонность и долгосрочный тренд, что ограничивает его качество.

Ridge Regression на признаках (B3) показала лучший результат благодаря грамотному инжинирингу признаков. Лаги улавливают автокорреляцию, rolling статистики отражают локальную волатильность и тренд, а календарные признаки (особенно hour_cos и day_of_week) эффективно моделируют суточные и недельные циклы. Коэффициент rolling_mean_14 = 11.08 указывает на сильную зависимость от долгосрочного среднего, что отражает восходящий тренд.

GRU (R1) показала результат хуже Ridge Regression, несмотря на способность автоматически извлекать временные зависимости. Возможные причины: относительно небольшой объем обучающих данных (2554 последовательности), GRU чувствительна к выбору гиперпараметров и могла переобучиться (лучшая эпоха всего 6-я), а также для данного ряда линейные зависимости оказались достаточными.

Тренд и суточная сезонность очень заметны в данных, что подтверждается важностью признаков rolling_mean_14 (тренд) и hour_cos (суточный цикл). Именно эти компоненты позволили Ridge Regression так точно прогнозировать.

Потенциальные утечки данных могли возникнуть при: 1) использовании future rolling (среднее по будущим значениям) — мы использовали min_periods=1, что безопасно; 2) случайном перемешивании перед split — мы использовали temporal split; 3) масштабировании на всех данных — мы обучали scaler только на train.

Выбор по validation, а не по test, принципиален, так как test должен оставаться "нетронутым" до финальной оценки. Использование test для выбора модели приводит к подгонке под тестовые данные и завышению качества.

Наиболее показательные ошибки модели (B3) пришлись на резкие пиковые значения (например, в районе 210) и моменты быстрого изменения тренда, что ожидаемо для нестационарного ряда.

## 8. Итоговый вывод

Для данной задачи в качестве базового подхода я бы выбрала Ridge Regression с правильно подобранными признаками, так как он показал наилучшее качество, интерпретируем и не требует больших вычислительных ресурсов. Простого baseline на признаках достаточно, когда ряд имеет четкую структуру (тренд + сезонность), которая хорошо описывается линейными комбинациями исторических значений. К GRU или другим нейросетям имеет смысл переходить при наличии более сложных нелинейных зависимостей, долгосрочных паттернов, которые не улавливаются линейными моделями, а также при большем объеме данных. Главный вывод о валидации временных рядов: temporal split — это не рекомендация, а обязательное требование, только хронологическое разбиение позволяет честно оценить способность модели прогнозировать будущее на основе прошлого.
