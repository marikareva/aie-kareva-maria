# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний:** 15 документов по машинному обучению, NLP, нейронным сетям и retrieval-системам. Тематика выбрана из-за четких определений и концепций, которые удобно извлекать по запросам.
- **Чанкинг:** Реализован класс `SimpleChunker` с разбиением по предложениям, параметры: `chunk_size=400`, `chunk_overlap=50`. Получено 37 чанков.
- **Embedding-модель:** TF-IDF векторизация + TruncatedSVD до 64 измерений с нормализацией для косинусного сходства.
- **Базовая проверка retrieval:** На 10 контрольных запросах достигнуты метрики: Hit@1=0.7, Hit@3=0.9, Hit@5=1.0, Recall@5=1.0.
- **Обновление базы знаний:** Добавлено 4 новых документа (LoRA, квантование, оценка эмбеддингов, PEFT). После переиндексации результаты поиска изменились для всех тестовых запросов.
- **Mini-RAG:** Реализован класс `MiniRAG` с методами retrieve и generate_answer. Система успешно отвечает на вопросы, возвращая источники.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Embedding-модель:** TF-IDF с параметрами `max_features=1000`, `ngram_range=(1,2)` + TruncatedSVD до 64 компонент. Векторы нормализованы для косинусного сходства.
- **Метрика сходства:** Косинусное сходство (через Inner Product после нормализации).
- **Индекс FAISS:** `IndexFlatIP` (точный поиск по внутреннему произведению).
- **top_k по умолчанию:** 5
- **Первые примеры retrieval:** На запросы "Что такое эмбеддинги?", "Как работает FAISS?", "Что такое RAG?" система уверенно находила релевантные документы с высокими score.

### 4.2. Контрольные запросы и оценка retrieval

- **Контрольных запросов:** 10
- **Что считалось релевантным:** Соответствие заголовка найденного документа ожидаемому `doc_id` из разметки.
- **Метрики retrieval:** Hit@k, Recall@k, MRR@k для k=1,3,5.
- **Итоговый hit@k:** Hit@1=0.7, Hit@3=0.9, Hit@5=1.0
- **Итоговый recall@k:** Recall@1=0.5, Recall@3=0.8, Recall@5=1.0
- **Дополнительные метрики:** MRR@5≈0.85
- **Самые простые запросы:** "Что такое машинное обучение?", "Какие бывают индексы в FAISS?" (релевантный документ на 1 месте).
- **Самые проблемные запросы:** "Чем BERT отличается от обычных эмбеддингов?" (требует сравнения, релевантные документы BERT и SBERT находятся, но не на первом месте).

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Какой параметр сравнивался:** `top_k` (количество возвращаемых документов).
- **Какие варианты сравнивались:** k=1, k=3, k=5.
- **Как изменилось качество:** Hit@k вырос с 0.7 (k=1) до 1.0 (k=5), Recall@k с 0.5 до 1.0.
- **Какой вариант выбран как основной:** k=3 как баланс между полнотой и объемом контекста для RAG.
- **Что обновлено в базе знаний:** Добавлено 4 новых документа: LoRA, квантование, оценка качества эмбеддингов, PEFT.
- **Как выполнена переиндексация:** Повторный чанкинг объединенной базы (19 документов), переобучение эмбеддера, создание нового индекса FAISS.
- **Что изменилось после обновления:** Для запросов "Что такое LoRA?", "Расскажи про квантование", "Как оценить качество эмбеддингов?", "Какие есть методы PEFT?" до обновления находились нерелевантные документы, после — целевые новые документы на первых позициях.

## 7. Анализ

Выбранная база знаний оказалась удачной для учебных целей: 15 документов достаточно для демонстрации retrieval, а тематика ML/NLP позволяет формулировать естественные запросы. Чанкинг с параметрами 400/50 дал 37 фрагментов — достаточно для точного поиска без излишней фрагментации. Увеличение overlap помогло сохранить контекст на границах чанков.

Формальная оценка retrieval на контрольных запросах показала высокое качество: Hit@5=1.0, Recall@5=1.0. Это подтверждает, что векторный поиск эффективно находит релевантные документы. Эксперимент с k продемонстрировал ожидаемый рост метрик при увеличении k, что важно учитывать при настройке RAG.

Обновление базы знаний показало, что система корректно адаптируется к новым данным после переиндексации. Новые документы успешно находились по соответствующим запросам, что подтверждает работоспособность пайплайна обновления.

Наиболее показательные ошибки mini-RAG связаны с запросами на сравнение. Система находит документы по отдельным сущностям (BERT, SBERT), но не может синтезировать сравнительный ответ, так как в базе нет документа с прямым сравнением. Это демонстрирует ограничение retrieval-based подхода: ответ ограничен содержанием базы знаний.

Ограничения учебного решения: используется упрощенный эмбеддер (TF-IDF+SVD), шаблонная генерация ответов вместо LLM, нет реранкинга. Однако для демонстрации принципов работы FAISS и mini-RAG этого достаточно. Использование GPU (CUDA) ускорило вычисления, хотя для TF-IDF+SVD это не критично — основное преимущество GPU проявилось бы при использовании transformer-моделей.

## 8. Итоговый вывод

В качестве базового retrieval-конвейера для такой задачи я бы оставил `chunk_size=400`, `overlap=50`, `k=3` и `IndexFlatIP` с нормализованными векторами. Это дает хороший баланс точности и полноты.

Главное, что я понял про качество retrieval: даже простые эмбеддинги (TF-IDF+SVD) могут давать высокие метрики на небольших тематических базах, но для production нужны более сильные модели (SBERT, OpenAI embeddings). Наличие GPU позволяет в будущем легко перейти на transformer-модели.

Главное про связь retrieval и mini-RAG: качество ответа критически зависит от качества retrieval. Если релевантный документ не попал в top-k, ответ будет неверным. Поэтому оценка retrieval метриками (Hit@k, Recall@k) — обязательный этап перед внедрением RAG.
