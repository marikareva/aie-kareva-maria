**Краткое описание:**

Проект представляет собой end-to-end сервис для прогнозирования задержек авиарейсов. По характеристикам рейса (время вылета, день недели, аэропорт, авиакомпания, расстояние) модель предсказывает задержку в минутах. Сервис реализован на FastAPI, упакован в Docker, покрыт тестами и готов к развёртыванию.

## 2. Постановка задачи и контекст

**Предметная область и задача:**

Задача — регрессия: предсказать задержку вылета рейса в минутах. Потенциальные пользователи сервиса:

- **Авиакомпании** — для оперативного управления расписанием и информирования пассажиров.
- **Аэропорты** — для прогнозирования нагрузки на выходы, персонал и обработку багажа.
- **Пассажиры** — для оценки риска опоздания на стыковочный рейс.

Сценарий использования: пользователь (или система) отправляет параметры рейса через API и получает прогноз задержки с категорией (без задержки, умеренная, серьёзная).

**Формулировка задачи в терминах ML:**

- **Входные данные:** 9 признаков рейса (час вылета, день недели, месяц, флаг выходного, расстояние, лаг-задержка аэропорта, загруженность аэропорта, средняя задержка авиакомпании).
- **Целевая переменная:** `DEPARTURE_DELAY` — задержка вылета в минутах (клиппирована в диапазон [-60, 180]).
- **Ограничения:** время ответа API < 200 мс, модель должна работать на CPU.

**Целевые метрики качества:**

- **MAE (Mean Absolute Error)** — средняя абсолютная ошибка в минутах. Основная метрика, так как она интерпретируема бизнесом: «модель ошибается в среднем на X минут».
- **R² (коэффициент детерминации)** — доля объяснённой дисперсии. Дополнительная метрика для оценки качества регрессии.
- **Доля предсказаний с ошибкой ≤ 15 минут** — бизнес-интерпретируемая метрика: «в каком проценте случаев модель предсказала достаточно точно».

## 3. Данные

**Источник данных:**

Синтетический датасет (50 000 записей), сгенерированный с учётом реалистичных закономерностей авиаперевозок. Данные моделируют работу крупных аэропортов (JFK, LAX, ORD, ATL и др.) и авиакомпаний (Delta, American, Southwest и др.). Для production-использования датасет может быть заменён на реальный (например, US DOT Flight Delays Dataset с Kaggle).

**Структура данных:**

Файл `data/raw/flights.csv`. Ключевые признаки:

- **Числовые:** `DISTANCE` (расстояние), `DEPARTURE_DELAY` (целевая переменная), `DEP_HOUR` (час вылета).
- **Категориальные:** `AIRLINE` (код авиакомпании), `ORIGIN_AIRPORT` (аэропорт вылета), `DESTINATION_AIRPORT` (аэропорт прилёта).
- **Временные:** `YEAR`, `MONTH`, `DAY`, `DAY_OF_WEEK`.
- **Бинарные:** `CANCELLED` (отмена рейса).

**Предобработка и EDA:**

Основные шаги предобработки:
1. Удаление отменённых рейсов (`CANCELLED == 1`).
2. Заполнение пропусков в `DEPARTURE_DELAY` нулями.
3. Циклическое кодирование часа вылета (sin/cos) для сохранения близости значений 23:00 и 00:00.
4. Создание лаг-признаков: средняя задержка последних 3 рейсов из аэропорта.
5. Аггрегация: загруженность аэропорта (число рейсов в день), средняя задержка авиакомпании.

Важные наблюдения из EDA:
- Распределение задержек скошено вправо: большинство рейсов без задержек, но есть длинный хвост опозданий.
- Вечерние рейсы (16–20 часов) задерживаются сильнее утренних — эффект накопления задержек.
- В выходные дни задержки меньше.
- Авиакомпании заметно различаются по средней задержке.

Графики EDA сохранены в `notebooks/eda_plots.png`. Пайплайн предобработки реализован в `src/eda_and_train.py`.

## 4. Модели и подходы

**Базовые (baseline) модели:**

В качестве baseline использована **LinearRegression** — простая линейная модель. Она показала следующие результаты на тестовой выборке:
- MAE: 7.1 мин
- R²: 0.429
- Доля предсказаний ±15 мин: 88.8%

Baseline выбран для оценки нижней границы качества и проверки, насколько данные линейно разделимы.

**Улучшенные модели и эксперименты:**

Поверх baseline обучен **CatBoostRegressor** — градиентный бустинг, устойчивый к выбросам и разнотипным признакам. Параметры:
- iterations: 500
- learning_rate: 0.05
- depth: 6
- early_stopping_rounds: 50

Результаты CatBoost:
- MAE: 7.1 мин
- R²: 0.424
- Доля предсказаний ±15 мин: 88.9%

**Сравнение:**

Обе модели показали близкие результаты. На синтетических данных зависимости оказались близки к линейным, поэтому простая модель не уступила градиентному бустингу. На реальных данных с нелинейными взаимодействиями (погода × время суток, сезонность) CatBoost ожидаемо покажет преимущество.

Все эксперименты и код обучения — в `src/eda_and_train.py`.

## 5. Экспериментальный протокол и результаты

**Экспериментальный протокол:**

- Выборка разделена на train/test в пропорции 80/20 (`random_state=42`).
- Метрики рассчитаны на тестовой выборке.
- Для CatBoost использован early stopping на 50 раундах для предотвращения переобучения.

**Сравнение моделей:**

| Модель | MAE (мин) | R² | ±15 мин |
|--------|:---------:|:---:|:-------:|
| LinearRegression (Baseline) | 7.1 | 0.429 | 88.8% |
| CatBoostRegressor | 7.1 | 0.424 | 88.9% |

**Выбор финальной модели:**

Автоматически выбрана модель с лучшим MAE. На синтетических данных это LinearRegression. Модель проста, быстра и интерпретируема. При замене данных на реальные пайплайн автоматически выберет лучшую модель заново.

## 6. Архитектура решения и сервис

**Архитектура пайплайна:**

Данные проходят путь:
1. `data/raw/flights.csv` → `src/eda_and_train.py` → `models/model.pkl` (обучение)
2. Модель загружается в `src/app/api.py` (FastAPI-сервис)
3. Пользователь отправляет запрос → сервис возвращает предсказание

**API и endpoints:**

- `GET /health` — проверка здоровья сервиса, статус загрузки модели
- `POST /predict` — предсказание задержки по параметрам рейса

**Параметры `/predict` (JSON):**
- `dep_hour` (int, 0–23) — час вылета
- `day_of_week` (int, 0–6) — день недели
- `month` (int, 1–12) — месяц
- `is_weekend` (int, 0–1) — флаг выходного
- `distance` (float, >0) — расстояние
- `airport_lag_delay` (float) — лаг-задержка аэропорта
- `airport_load` (float) — загруженность аэропорта
- `airline_avg` (float) — средняя задержка авиакомпании

**Ответ:**
{
  "predicted_delay_minutes": 15.7,
  "is_delayed": true,
  "delay_category": "Умеренная задержка",
  "timestamp": "2026-06-01T12:00:00"
}

**Технологический стек:**
- Python 3.12, FastAPI, Pydantic, uvicorn
- CatBoost, scikit-learn, pandas, numpy
- pytest (тесты), Docker (контейнеризация)

**Команды для запуска:**
uvicorn src.app.api:app --reload                  # локально
docker build -t flight-delay-api .                # сборка образа
docker run -p 8000:8000 flight-delay-api          # запуск в контейнере

## 7. Наблюдаемость, конфигурация и безопасность

**Логи и наблюдаемость:**

- Логируются все запросы: метод, URL, статус ответа, время обработки.
- Логи пишутся в файл `api.log` и выводятся в консоль.
- Реализован middleware для замера времени ответа каждого запроса.
- Эндпоинт `/health` позволяет проверить, что сервис запущен и модель загружена.

**Конфигурации:**

- Настройки вынесены в `configs/config.json` (пути к модели, параметры сервера, уровень логирования).
- Шаблон переменных окружения — `configs/.env.example` (без реальных секретов).
- Реальный `.env` добавлен в `.gitignore` и не попадает в репозиторий.

**Безопасность:**

- В репозитории нет токенов, паролей, ключей API.
- Все секреты вынесены в `.env` (игнорируется Git).
- `data/raw/` содержит только синтетические данные, не содержащие персональной информации.

## 8. Ограничения и дальнейшая работа

**Ограничения:**

- Использованы синтетические данные — зависимости могут отличаться от реальных.
- Не учтены погодные условия — один из ключевых факторов задержек.
- Модель обучена на статическом датасете — не обновляется онлайн.
- Нет мониторинга дрейфа данных и модели в production.

**Дальнейшая работа:**

- **Модель:** заменить синтетику на реальный датасет (US DOT Flight Delays), добавить погодные признаки через API, обучить более сложные архитектуры.
- **Сервис:** добавить `/batch_predict` для пакетной обработки, prometheus-метрики для мониторинга, кэширование частых запросов.
- **Данные и эксперименты:** настроить MLflow для трекинга экспериментов, добавить кросс-валидацию, исследовать feature importance.
